# 02-modules / i18n — The `:i18n` module

> Shared string resources for the app, inherited from Mihon. Built with
> [Moko Resources](https://github.com/icerockdev/moko-resources) so the same
> catalog compiles for both Kotlin Multiplatform common code and Android. Original
> English strings live in `base/`; translations are produced externally on Weblate
> and merged back here.

## Purpose

`/home/z/.../ANIYOMI/i18n/` is a small Kotlin Multiplatform library whose only
content is localised strings (and a few plurals). It exposes a generated accessor
object — `tachiyomi.i18n.MR` — that the rest of the project uses to look up
strings without touching the Android `R` class directly. This is the same setup
Mihon uses; Aniyomi keeps the inherited catalog unchanged and layers its own
additions in the separate [`:i18n-aniyomi`](i18n-aniyomi.md) module.

## Build configuration

Source: `../ANIYOMI/i18n/build.gradle.kts`.

```kotlin
plugins {
    id("mihon.library")
    kotlin("multiplatform")
    alias(libs.plugins.moko)            // dev.icerock.mobile.multiplatform-resources
}

kotlin {
    androidTarget()
    applyDefaultHierarchyTemplate()
    sourceSets {
        commonMain { dependencies { api(libs.moko.core) } }
    }
    compilerOptions { freeCompilerArgs.add("-Xexpect-actual-classes") }
}

android {
    namespace = "tachiyomi.i18n"
    sourceSets {
        val main by getting
        main.res.srcDirs("src/commonMain/resources", generatedAndroidResourceDir)
    }
    lint { disable.addAll(listOf("MissingTranslation", "ExtraTranslation")) }
}

multiplatformResources {
    resourcesPackage.set("tachiyomi.i18n")   // → generates tachiyomi.i18n.MR
}
```

Key points:

- **Plugin `mihon.library`** — the shared library convention plugin (see
  [`../00-overview/04-build-system.md`](../00-overview/04-build-system.md)).
- **`kotlin("multiplatform")` + `androidTarget()`** — KMP module with a single
  target (Android). The `commonMain` source set is where the resources live.
- **`alias(libs.plugins.moko)`** — the Moko Resources Gradle plugin
  (`dev.icerock.mobile.multiplatform-resources`); it scans
  `src/commonMain/moko-resources/` and generates the `MR` accessor.
- **`multiplatformResources.resourcesPackage`** — controls the package of the
  generated `MR` object. Here it is `tachiyomi.i18n`, so the accessor is
  `tachiyomi.i18n.MR`.
- **`lint.disable += MissingTranslation/ExtraTranslation`** — translations are
  maintained externally on Weblate, so missing/extra keys per-locale are not
  treated as build-breaking lint errors.
- **`generateLocalesConfig` task** — wires the `LocalesConfigTask` from
  `buildSrc/src/main/kotlin/mihon/buildlogic/tasks/LocalesConfigTask.kt`. It
  scans `src/commonMain/moko-resources/**/strings.xml`, filters out empty
  catalogs, and emits `xml/locales_config.xml` into the generated Android
  resource dir so Android 13+ can advertise supported locales.

## Moko Resources structure

```
i18n/
├── build.gradle.kts
├── README.md
└── src/
    ├── main/
    │   └── AndroidManifest.xml         (empty <manifest/> — library only)
    └── commonMain/
        └── moko-resources/
            ├── base/                   ← English source strings (the "default")
            │   ├── strings.xml
            │   └── plurals.xml
            ├── ar/                     ← Arabic
            │   ├── strings.xml
            │   └── plurals.xml
            ├── zh-rCN/                 ← BCP-47 with region prefix
            ├── pt-rBR/
            ├── ka-rGE/
            └── …                       (one folder per locale)
```

Conventions:

- Each locale has its own folder named with the BCP-47 code (with `-r` for
  region variants, e.g. `zh-rCN`, `pt-rBR`, `ka-rGE`).
- Each folder contains a `strings.xml`. Most also contain a `plurals.xml`.
- The `base/` folder is the source-of-truth English catalog. The
  `generateLocalesConfig` task maps `base` → `en` when emitting
  `locales_config.xml`.
- Empty `<resources/>` files are tolerated (and skipped by the locales task)
  so translators can stub a locale before filling it in.

## Locale count

There are **67 locale folders** under `i18n/src/commonMain/moko-resources/`
(including `base`). That is **66 translated locales** plus the English source
catalog. The list (alphabetical):

```
am ar as base be bg bn ca ceb cs cv da de el eo es eu fa fi fil fr gl he hi
hr hu in it ja jv ka-rGE kk km kn ko lt lv ml mr ms my nb-rNO ne nl nn pl pt
pt-rBR ro ru sa sah sc sdh sk sq sr sv ta te th tr uk uz vi zh-rCN zh-rTW
```

(Note: `in` is the legacy Java code for Indonesian; `nb-rNO` is Norwegian
Bokmål; `pt-rBR` is Brazilian Portuguese, parallel to the `pt` European
catalog.)

## String catalog size

The English `base/strings.xml` is **986 lines / 847 `<string>` entries**. It
is organised with comment banners (`<!-- … -->`) into sections:

- Generic strings, Models, Screen titles, Shared labels, Actions
- Shortcuts, Onboarding
- Preferences (General / Library / Extension / Reader / Downloads / Tracking /
  Browse / Data and storage / Sync / Advanced / About / ACRA)
- More, Login dialog
- Library, Browse source, Manga info, Manga chapters, …
- `base/plurals.xml` is 98 lines / a handful of `<plurals>` entries (counts,
  "N items selected", etc.).

The keys cover everything inherited from Mihon: manga library, reader,
downloads, trackers, backup, settings. Aniyomi-specific additions (player,
anime UI, hosters, torrents) live in [`:i18n-aniyomi`](i18n-aniyomi.md).

## How `:app` consumes `:i18n`

`/home/z/.../ANIYOMI/app/build.gradle.kts` declares:

```kotlin
implementation(projects.i18n)
implementation(projects.i18nAniyomi)
```

At runtime, code anywhere in `:app` imports and references the generated
`MR` object directly:

```kotlin
import tachiyomi.i18n.MR
// …
stringResource(MR.strings.label_library)
MR.strings.action_filter_started.getString(context)
```

`MR.strings.<key>` returns a `StringResource` that the Compose `stringResource(...)`
helper (or the Moko `getString(context)` extension) resolves against the current
locale at draw time. Hundreds of files under
`app/src/main/java/eu/kanade/` import `tachiyomi.i18n.MR`; a representative
example is
`../ANIYOMI/app/src/main/java/eu/kanade/presentation/browse/manga/MigrateMangaScreen.kt`.

Several other modules also pull `:i18n` in via `api(projects.i18n)` so the `MR`
accessor is visible transitively:

- `:core:common` (`implementation`)
- `:source-local` (`api`)
- `:presentation-core` (`api`)
- `:presentation-widget` (`api`)

This is why feature modules can reference `MR.strings.*` without re-declaring
the dependency.
